# Plan: 타이머기능

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 사용자가 특정 시간대에만 패킷 캡처를 자동으로 시작/종료하려면 수동으로 직접 실행해야 한다. |
| **Solution** | 시작 시각과 종료 시각을 입력받아 해당 시간대에만 캡처를 자동 실행하는 타이머 기능을 추가한다. |
| **Functional UX Effect** | 사용자는 캡처 시작 전에 타이머를 설정하면 지정 시각에 자동으로 캡처가 시작되고 종료 시각에 자동 종료된다. |
| **Core Value** | 야간/특정 시간대 모니터링 자동화로 사용자가 자리를 비워도 원하는 시간대 트래픽만 선택적으로 수집할 수 있다. |

---

## 1. 배경 및 목적

goscope는 현재 실행 즉시 패킷 캡처를 시작한다. 사용자가 특정 시간대(예: 업무 시간, 야간)에만 캡처를 원할 경우, 직접 그 시각에 실행해야 하는 불편함이 있다. 타이머 기능을 추가하여 스케줄 기반 캡처를 지원한다.

---

## 2. 기능 요구사항

### 2.1 핵심 기능

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F-01 | 시작 시각 입력 (HH:MM, 생략 시 즉시 시작) | Must |
| F-02 | 종료 시각 입력 (HH:MM) | Must |
| F-03 | 시작 시각까지 대기 후 캡처 자동 시작 | Must |
| F-04 | 종료 시각에 캡처 자동 종료 | Must |
| F-05 | 자정을 넘기는 시간대 지원 (예: 23:00~01:00) | Should |
| F-06 | 백그라운드 모드와 타이머 조합 지원 | Should |

### 2.2 입력 흐름

```
BPF 필터 입력
      |
타이머를 설정하시겠습니까? [y/N]
  -> N: 즉시 실행 (기존 동작)
  -> Y:
       시작 시각 (예: 14:00, 생략 시 즉시 시작):
       종료 시각 (예: 15:00):
            |
       백그라운드 여부 (RotateDir 있는 경우)
```

### 2.3 동작 규칙

- 시작 시각이 현재보다 이전이면 다음날로 자동 설정
- 종료 시각이 시작 시각보다 이전이면 다음날로 처리 (자정 넘기는 케이스)
- 시작 시각 생략 시 즉시 캡처 시작, 종료 시각만 적용
- 대기 중 Ctrl+C 입력 시 정상 종료

## 3. 비기능 요구사항

- 기존 Config 직렬화(JSON) 호환 유지 (백그라운드 데몬 전달)
- 플랫폼 무관 동작 (Linux, macOS, Windows)

---

## 4. 구현 범위

### 변경 파일

| 파일 | 변경 내용 |
|------|-----------|
| `internal/cli/prompt.go` | Config에 TimerStart/TimerEnd 추가, promptTimer() 함수 추가 |
| `internal/cli/runner.go` | 타이머 대기(time.Sleep) 및 context.WithDeadline 적용 |

### 변경하지 않는 파일

- `internal/capture/` — 캡처 엔진 무변경
- `internal/daemon/` — 데몬 로직 무변경 (Config JSON 직렬화로 자동 전달)
